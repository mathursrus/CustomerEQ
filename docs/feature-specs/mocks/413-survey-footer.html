Footer is neutral and byte-identical across all four states (per R12 + #378 NFR-S5 timing-attack-resistance against token enumeration).

R12 + #378 implication: the footer DOM and bytes must be identical across expired / responded / survey-not-open / invalid — any per-state footer variation would leak the state to an attacker timing the response. Card chrome above the footer can vary by state because the server returns the same uniform {state} payload for all four; chrome differences are rendered client-side and don't appear in the response bytes.

Convergence (per #476): the responded card here and Scene 3's amber "Already responded" card should both be the same SurveyStateCard variant="already-responded" with chrome theming as the only difference (white here for timing-attack uniformity, amber for the direct-link path).
